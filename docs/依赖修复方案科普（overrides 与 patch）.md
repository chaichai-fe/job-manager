# 依赖修复方案科普：`overrides` 锁版本 vs `patch` 打补丁

> 纯技术科普，不绑定任何具体项目。讲清楚当第三方依赖「不对劲」时，你手里有哪些武器、各自改的是什么、什么时候用哪个、以及坑在哪。

---

## 0. 先理解问题：依赖为什么会「不对劲」

你装了一个 npm 包，结果它在你的环境里表现不正常。常见就三类原因：

1. **版本冲突**：包 A 依赖 `lib@^2`，包 B 依赖 `lib@^3`，两个大版本行为不兼容，安装器解析出来的那一个把另一边搞崩了；或者某个间接依赖偷偷升了一个 minor，引入了 break。
2. **包本身有 bug / 不兼容你的运行环境**：源码里用了你的环境没有的 API（比如在边缘运行时里用了 Node 专有 API），或者就是有个明确的 bug，官方还没修没发版。
3. **你想改它的行为**：删掉一段不想要的副作用、加一行日志排查、临时改个默认值。

针对这些，社区演化出两大类「不 fork、不等官方」的手段，区别就一句话：

| 手段 | 改的是什么 | 一句话 |
| --- | --- | --- |
| **`overrides` / `resolutions`**（锁版本） | 改「**用哪个版本**」 | 强制整棵依赖树都用你指定的某个**已存在**的版本 |
| **`patch`**（打补丁） | 改「**包里的代码**」 | 把你对某个版本源码的改动，固化成一个 diff 文件，安装时自动重新打上 |

**核心判断**：问题能靠「换一个已发布的版本」解决吗？能 → `overrides`；不能（必须改代码本身）→ `patch`。

---

## 1. `overrides` 方案（锁/改版本）

### 1.1 它是什么

在 `package.json`（或对应配置）里声明一条规则：**不管依赖树里谁要求什么版本，某个包一律给我用这个版本。** 它不碰任何源码，只是干预「依赖解析（resolution）」这一步。

各包管理器的字段名不同，机制一致：

```jsonc
// npm（package.json）
{
  "overrides": {
    "lib": "1.2.3"
  }
}
```

```jsonc
// pnpm（package.json）
{
  "pnpm": {
    "overrides": {
      "lib": "1.2.3"
    }
  }
}
```

```jsonc
// Yarn (berry / classic) —— 叫 resolutions
{
  "resolutions": {
    "lib": "1.2.3"
  }
}
```

```jsonc
// Bun（package.json）—— 也叫 overrides
{
  "overrides": {
    "lib": "1.2.3"
  }
}
```

> 注：Yarn 历史上用 `resolutions`，npm/pnpm/Bun 用 `overrides`，含义基本等价。下文统称「overrides」。

### 1.2 进阶写法

不只是「钉死一个版本」，还能更精细：

```jsonc
{
  "overrides": {
    // 只在 "pkgA" 的子依赖里把 lib 钉到 1.2.3，别处不动
    "pkgA": {
      "lib": "1.2.3"
    },
    // 把 lib 强行指向另一个包 / 某个 git 版本 / 本地路径
    "lib": "npm:lib-fork@1.2.3",
    // 引用当前直接依赖里某个版本（npm 的 $ 语法）
    "other": "$other"
  }
}
```

### 1.3 什么时候用

- **间接依赖升级引入了 bug**：把它钉回上一个好版本。
- **解决「多版本打架」**：强制全树统一到一个兼容版本，去重、减体积、消歧义。
- **上游已经在某个版本修了**，但你的依赖树因为版本范围没解析到它 → 直接钉到修好的那个版本。
- **临时挡住一个有漏洞的传递依赖**：在官方链路更新前先锁到安全版本。

### 1.4 优点 / 缺点

| 优点 | 缺点 / 风险 |
| --- | --- |
| 零侵入，不改源码，最干净 | **只能用「已经存在」的版本**，改不了源码本身 |
| 配置一行，所有包管理器原生支持 | 强行降/换版本可能与依赖它的包**不兼容**（API 对不上） |
| 升级、维护成本低 | 钉死后容易**长期遗忘**，挡住了本该有的安全/功能更新 |
| 对「版本冲突」类问题是首选 | 对「包源码就是错的 / 缺功能」无能为力 |

### 1.5 注意事项

- overrides 是**强制**的，会越过包自己声明的版本范围，可能把某个包配上它根本没测过的依赖版本，务必跑测试验证。
- 写清楚**为什么钉**（注释或 commit 说明 + 关联 issue），否则半年后没人敢动。
- 定期复查：上游正常修复 / 发版后，应及时移除 override，避免长期偏离主线。

### 1.6 `overrides` 和 `dependencies` 里的版本号有什么区别

很容易混的一点：两者都写「包名 + 版本」，但**作用范围和语义完全不同**。

| | `dependencies`（及 `devDependencies`）里的版本 | `overrides` |
| --- | --- | --- |
| 管谁 | 只管**你直接安装、直接 `import` 的那一个包** | 管**整棵依赖树里任意层级**的包（含间接依赖） |
| 语义 | 「我想要这个版本」——是**建议范围**，`^` / `~` 允许浮动 | 「必须是这个版本」——是**强制命令**，越过别人声明的范围 |
| 管得到间接依赖吗 | ❌ 管不到。间接依赖的版本由引入它的那个包自己决定 | ✅ 管得到。这正是它存在的意义 |
| 版本号写法 | 常带 `^`，允许小版本自动升级 | 通常写**精确版本**，就是要钉死 |
| 典型用途 | 声明项目正常用到的库 | 修复 / 兜底间接依赖的版本问题 |

**关键点**：`dependencies` 里的版本号**只约束这一个包自己**，它**管不到「依赖的依赖」**。比如包 A 内部又依赖了 `lib`，那个 `lib` 装什么版本是由 A 决定的，你在自己的 `dependencies` 里写 `lib` 也不一定能改到 A 用的那一份。要强行统一/钉死间接依赖，才需要 `overrides`。

举个本项目里的真实例子：`kysely` **并没有出现在 `dependencies` 里**，它是被 `better-auth` 等包间接拉进来的。所以想锁它的版本，只能用 `overrides`，写进 `dependencies` 是无效的：

```jsonc
{
  "dependencies": {
    "better-auth": "^1.6.14"   // 直接依赖；管不到它内部用的 kysely
  },
  "overrides": {
    "kysely": "0.28.17"        // 强制整棵树的 kysely 都用这个版本（含间接依赖）
  }
}
```

> 一句话：`dependencies` = 「我直接要装什么」（只管自己这一层、可浮动）；`overrides` = 「整棵依赖树里某个包必须是哪个版本」（连别人的依赖也强制改、用来打补丁）。

## 2. `patch` 方案（打补丁）

### 2.1 它是什么

`node_modules` 是「构建产物」，不进版本控制，任何一次 `install`/CI 都会重新拉取并覆盖。所以**手改 `node_modules/some-pkg/dist/index.js` 是无效的**：

- 队友、CI 拉不到，只在你本地生效；
- 下次 `install` / 升级时被无声覆盖，问题悄悄复活。

patch 机制解决的就是这件事：把你对某个包**某个版本**源码的改动，**记录成一个 diff 文件**（提交进 git），并在每次安装时**自动重新打到 `node_modules`** 上。这样改动可复现、可共享、可追溯。

### 2.2 各包管理器怎么做

**核心三步通用**：进入可编辑状态 → 改文件 → 生成补丁。

```bash
# Bun（内置）
bun patch <pkg>[@version]      # 解包到可写临时目录，终端会提示路径
# ... 编辑提示路径下的文件（改的是 dist/ 里编译后的产物）...
bun patch --commit <pkg>       # 生成 patches/ 下的 .patch，并写入 package.json

# pnpm（内置）
pnpm patch <pkg>               # 输出一个临时目录
# ... 改临时目录里的文件 ...
pnpm patch-commit <临时目录路径>  # 生成补丁并写入 package.json 的 pnpm.patchedDependencies

# Yarn berry（内置）
yarn patch <pkg>
# ... 改 ...
yarn patch-commit -s <路径>

# npm（无内置，用第三方 patch-package）
#   手改 node_modules/<pkg>/...
npx patch-package <pkg>        # 生成 patches/<pkg>+version.patch
#   再在 package.json 的 postinstall 脚本里加 "patch-package"，让每次安装自动应用
```

补丁产物（以 Bun/pnpm 为例）：

- 项目根多出 **`patches/` 目录**，里面是 `<pkg>@<version>.patch` 之类的 diff 文件；
- `package.json` 多出 `patchedDependencies`（pnpm 是 `pnpm.patchedDependencies`）字段，指向该补丁；
- lockfile 里也会记录补丁。

> **`patches/` 必须进 git，不能被 `.gitignore` 忽略**，否则别人/CI 装完没补丁。

### 2.3 改的是「编译后产物」，不是源码 TS

发布到 npm 的通常是 `dist/` 下**已编译**的 JS / `.d.ts`，不是作者的 TS 源码。所以打补丁时：

- 先找到真正被 import 的入口（看该包 `package.json` 的 `main` / `module` / `exports`）；
- 改编译后的代码，必要时连压缩格式都得照着改；
- 改 `.d.ts` 只影响**编译期类型**，不改运行行为；改运行行为要找编译后的 JS。两者别混。

### 2.4 什么时候用

1. **修 bug**：包有明确 bug，上游没修/没发版，但你线上等着用。
2. **兼容运行环境**：包引用了当前环境没有的 API（Node 专有 API、`fs`、动态 `require` 等），小改让它在目标环境（如边缘运行时）跑通。
3. **打日志排查**：临时塞 `console.log` 定位问题（查完就删的补丁）。
4. **删多余副作用**：某包 import 时执行了不想要的初始化 / 遥测 / polyfill。
5. **改类型定义**：自带 `.d.ts` 有误，改一行让 TS 通过。
6. **安全热修**：依赖爆漏洞、官方补丁还没出，先本地补上兜住。

### 2.5 优点 / 缺点

| 优点 | 缺点 / 风险 |
| --- | --- |
| **能真正改源码**，overrides 做不到的它都能做 | **上游一升级，补丁极易对不上**，需重做 |
| 改动随仓库分发，队友/CI 自动应用，可复现 | 改的是编译产物，可读性差、定位费劲 |
| 适合「小改、急用、临时止血」 | 维护成本高于 overrides，容易变技术债 |
| 比 fork 整个仓库轻得多 | 大改 / 长期维护场景下不如直接 fork |

### 2.6 注意事项

- **补丁要尽量小**：只改必要几行，越小越不容易因上游变动冲突，review 也清楚。
- **写下「为什么」**：补丁旁/commit 注明原因 + 上游 issue 链接，方便日后判断「还需不需要」。
- **版本号写进文件名是有意为之**：升级被打补丁的包时，旧补丁多半打不上、会报警告；要么删（上游已修），要么对新版本重新走一遍。
- **CI 一致性**：确保 CI 跑了 `install` 并应用补丁（而非缓存了未打补丁的 `node_modules`）；frozen-lockfile 模式下补丁仍会应用，但 lockfile 必须已包含补丁记录。
- **定期回收**：每次升级依赖时检查 `patches/` 是否还需要，上游修了就删。

## 3. 横向对比

| 维度 | `overrides` / `resolutions` | `patch` |
| --- | --- | --- |
| 改的对象 | 依赖解析（用哪个版本） | 包的实际代码 |
| 能改源码吗 | ❌ 不能 | ✅ 能 |
| 能解决版本冲突吗 | ✅ 强项 | ❌ 不是为此设计 |
| 侵入性 | 极低（一行配置） | 中（diff 文件 + 安装钩子） |
| 维护成本 | 低 | 中—高（随上游升级失效） |
| 受上游升级影响 | 小（换个版本号即可） | 大（补丁常对不上要重做） |
| 典型适用 | 间接依赖坏了 / 多版本打架 / 锁安全版本 | 包有 bug / 不兼容环境 / 要删副作用 |
| 各管理器支持 | npm/pnpm/yarn/bun 原生 | bun/pnpm/yarn 内置；npm 靠 patch-package |

## 4. 怎么选：决策流程

```
依赖出问题
   │
   ├─ 换一个「已发布的版本」就能解决吗？
   │        │
   │        ├─ 能 ──► 用 overrides / resolutions 锁版本   ← 首选，最干净
   │        │
   │        └─ 不能（必须改包里的代码）
   │                 │
   │                 ├─ 改动小、急用、临时 ──► patch 打补丁
   │                 │
   │                 └─ 改动大、要长期维护 ──► fork 仓库自己维护（发私有包 / git 依赖）
   │
   └─ 无论用哪种：都视为「临时方案」，最终目标是推动上游修复或升级后撤掉。
```

经验法则：

- **优先级**：能 overrides 就别 patch，能 patch 就别 fork。越往后越重、维护成本越高。
- **两者可叠加**：常见组合是「`overrides` 把某依赖钉到一个稳定版本 + 对这个版本再 `patch` 一处小问题」，先锁住版本让补丁有稳定的施加目标。
- **都是止血**：overrides 和 patch 都是绕过上游的临时手段。长期正解永远是 **提 PR 给上游 / 等官方修复 / 升级**，然后把临时方案撤掉。

---

## 5. 与其他方案的全景对照

| 方案 | 适用 | 缺点 |
| --- | --- | --- |
| **等上游修 + 升级** | 长期正解 | 慢，可能不被合并 |
| **`overrides` / `resolutions`** | 只需换一个已存在的版本 | 改不了源码本身 |
| **`patch`** | 改动小、要立刻生效、想留在主仓库 | 上游一升级补丁可能失效，需重做 |
| **fork + 改 + 发私有包 / git 依赖** | 改动大、长期维护 | 重，需自己维护发布 |

---

## 一句话总结

> **改「用哪个版本」用 `overrides`，改「包里的代码」用 `patch`。** 前者解决版本冲突、零侵入、首选；后者能真正改源码、适合小改急用，但随上游升级易失效。两者都是临时止血，能叠加使用，最终都应被「上游修复 + 升级」取代。
