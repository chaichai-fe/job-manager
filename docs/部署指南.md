# 部署指南

本项目是一个 **TanStack Start** 应用，运行在 **Cloudflare Workers** 上，使用 **D1**（SQLite）做关系数据、**KV** 做会话存储。本文覆盖两类场景：

1. **首次部署**：从零创建云端资源，把项目第一次跑到线上。
2. **日常部署 / 更新**：代码或数据库 schema 变更后的再次发布。

---

## 0. 前置准备（一次性）

| 项              | 说明                                                                                 |
| --------------- | ------------------------------------------------------------------------------------ |
| Node.js         | **20.19+ 或 22.12+**（Vite 7 的最低要求，已放弃 Node 18；项目 `@types/node` 为 v22） |
| 包管理器        | `bun`（仓库自带 `bun.lock`）                                                         |
| Cloudflare 账号 | 需开通 Workers / D1 / KV（免费额度即可）                                             |
| Wrangler        | 项目内已作为 devDependency，无需全局安装，统一用 `bunx wrangler`                     |

登录 Cloudflare（首次会打开浏览器授权）：

```bash
bunx wrangler login
```

安装依赖：

```bash
bun install
```

> 关于依赖兼容：`better-auth` 用到的 kysely adapter 在**新版 `kysely`** 下会报 `MISSING_EXPORT`。本项目用 `package.json` 的 `overrides` 把 `kysely` 锁定到 `0.28.17` 来规避，安装时自动生效，**无需手动操作**。
> （项目当前**没有使用任何 patch**，`patches/` 目录也不存在；若将来确需改第三方包源码再考虑打补丁，方法见 [依赖打补丁（patch）指南](./依赖打补丁（patch）指南.md)。）

## 1. 首次部署

### 步骤 1 · 创建 D1 数据库

```bash
bunx wrangler d1 create job-manager-db
```

命令会返回一个 `database_id`，把它填入 `wrangler.jsonc`：

```jsonc
"d1_databases": [
  {
    "binding": "DB",
    "database_name": "job-manager-db",
    "database_id": "<这里填新返回的 id>",
    "remote": true,
    "migrations_dir": "drizzle/migrations"
  }
]
```

### 步骤 2 · 创建 KV 命名空间

```bash
bunx wrangler kv namespace create KV
```

把返回的 `id` 填入 `wrangler.jsonc`：

```jsonc
"kv_namespaces": [
  { "binding": "KV", "id": "<这里填新返回的 id>", "remote": true }
]
```

### 步骤 3 · 配置环境变量与密钥

`wrangler.jsonc` 的 `vars` 中改成你的线上域名（默认是占位域名）：

```jsonc
"vars": {
  "BETTER_AUTH_URL": "https://<你的 worker 域名>.workers.dev",
  "BETTER_AUTH_TRUSTED_ORIGINS": "https://<你的 worker 域名>.workers.dev"
}
```

> 域名通常是 `https://<worker-name>.<account-subdomain>.workers.dev`。第一次 `deploy` 后 Wrangler 会打印实际域名，可回来订正这两个值再部署一次。多个可信来源用英文逗号分隔。

设置生产环境的 better-auth 密钥（**只存在 Cloudflare，不进 git**）：

```bash
bunx wrangler secret put BETTER_AUTH_SECRET
```

可用 `openssl rand -base64 32` 生成一个随机值粘贴进去。

> 本地开发用的同名密钥写在 `.dev.vars`（已被 `.gitignore` 忽略），与线上的相互独立。

### 步骤 4 · 初始化线上数据库（应用迁移）

迁移文件在 `drizzle/migrations/`（已包含 `0000_init.sql`）。把它们应用到远端 D1：

```bash
bunx wrangler d1 migrations apply DB --remote
```

### 步骤 5 · 构建并发布

```bash
bun run deploy
```

`deploy` 脚本 = `vite build` + `wrangler deploy`。

完成后访问 Wrangler 打印的域名，注册第一个账号即可使用。

## 2. 日常部署 / 更新

### 场景 A：只改了代码（没动数据库 schema）

```bash
bun run deploy
```

### 场景 B：改了数据库 schema（`src/db/*.ts`）

1. 生成迁移 SQL：

```bash
bunx drizzle-kit generate
```

2. （推荐）先在本地验证：

```bash
bunx wrangler d1 migrations apply DB --local
bun run dev
```

3. 应用到线上并发布，一条命令搞定：

```bash
bun run deploy:full
```

`deploy:full` = `wrangler d1 migrations apply DB --remote` + `bun run deploy`。

> 也可以拆开执行：先 `bunx wrangler d1 migrations apply DB --remote`，再 `bun run deploy`。

## 3. 命令速查

| 命令                                            | 作用                                  |
| ----------------------------------------------- | ------------------------------------- |
| `bun install`                                   | 安装依赖（`overrides` 自动生效）      |
| `bun run dev`                                   | 本地开发，http://localhost:3000       |
| `bun run build`                                 | 仅构建（产物在 `dist/`）              |
| `bun run deploy`                                | 构建 + 发布到 Workers（**不**含迁移） |
| `bun run deploy:full`                           | 远端迁移 + 构建 + 发布                |
| `bunx drizzle-kit generate`                     | 由 schema 生成迁移 SQL                |
| `bunx wrangler d1 migrations apply DB --local`  | 应用迁移到本地 D1                     |
| `bunx wrangler d1 migrations apply DB --remote` | 应用迁移到线上 D1                     |
| `bunx wrangler secret put BETTER_AUTH_SECRET`   | 设置线上密钥                          |
| `bunx wrangler tail`                            | 查看线上实时日志                      |

## 4. 部署检查清单

首次上线前确认：

- [ ] `wrangler.jsonc` 里 `database_id`、KV `id` 已替换为真实值
- [ ] `vars` 里两个 URL 已改成线上域名
- [ ] 已 `wrangler secret put BETTER_AUTH_SECRET`
- [ ] 已对 `--remote` 执行过迁移
- [ ] `bun run deploy` 成功，域名可访问、可注册登录

## 5. 常见问题

- **依赖优化 / 打包报错**：删除 `node_modules/.vite` 缓存后重启 `bun run dev`。
- **`MISSING_EXPORT` 报错**：确认 `package.json` 的 `overrides`（`kysely: 0.28.17`）生效；删掉 `bun.lock` 与 `node_modules` 后重跑 `bun install` 让锁版本重新解析。
- **登录后立刻被踢回登录页**：多半是 `BETTER_AUTH_URL` / `BETTER_AUTH_TRUSTED_ORIGINS` 与实际访问域名不一致，订正 `vars` 后重新部署。
- **会话相关异常**：确认 KV 绑定正确（`binding: "KV"`），better-auth 的 session/限流数据存在 KV。
- **本地数据库为空**：首次本地开发需 `bunx wrangler d1 migrations apply DB --local` 初始化。
