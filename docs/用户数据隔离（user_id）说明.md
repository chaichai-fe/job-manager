# 用户数据隔离（`user_id`）说明

本文用大白话讲清楚「按 `user_id` 隔离」到底是什么、不做会怎样、以及为什么本项目这么做。结合 `server/*.ts` 与 `lib/server/auth-utils.ts` 的真实代码展开。

---

## 1. 前提：所有用户共用同一张表

这是理解隔离的关键前提。本项目的 `projects` / `project_statuses` / `repo_links` / `account_entries` **不是「每个用户一张表」**，而是**所有用户的数据都堆在同一张表里**，靠 `user_id` 这一列区分谁是谁（看板列定义 `project_statuses` 也按用户隔离——每个人有自己的一套状态列）：

| id  | user_id  | requirement | status  |
| --- | -------- | ----------- | ------- |
| a1  | 张三     | 做登录页    | done    |
| a2  | 张三     | 做支付      | todo    |
| a3  | **李四** | 改 bug      | testing |
| a4  | **李四** | 写文档      | todo    |

所谓「隔离」，就是：张三登录时只能看到 `user_id = 张三` 的两行，**绝对不能看到李四的数据**。

---

## 2. 不做隔离会怎样（后果很严重）

假设 `listProjects` 去掉了 `where` 过滤：

```ts
return db.select().from(projects) // ❌ 没有按 userId 过滤
```

那么**任何一个登录用户调用「查项目」，都会拿到全表数据**——张三能看到李四的所有项目、需求、备注。

更危险的是 `account_entries` 表里存着**明文密码**（用户自己记的测试账号密码）。一旦不隔离：

- 张三能看到所有人记的账号密码 → **数据泄露**
- 张三调 `deleteProject({ id: 'a3' })` 能删掉李四的项目 → **越权操作**

这类问题在安全里叫 **IDOR（Insecure Direct Object Reference，越权访问）**，是最常见也最致命的漏洞之一。

---

## 3. 为什么必须在「服务端」强制做，而不能信前端

直觉上也许会想：让前端传一个 `userId` 过来不就行了？**绝对不行。**

前端传来的任何东西都能被伪造——用户打开浏览器控制台，把请求里的 `userId: '张三'` 改成 `userId: '李四'`，就能读到李四的数据。

本项目的做法是：**`userId` 不从前端拿，而是服务端从「当前登录会话」里自己解析出来。**

```ts
// src/lib/server/auth-utils.ts
/** 要求登录，返回 userId，否则抛出 401。 */
export async function requireUserId(): Promise<string> {
  const user = await getCurrentUser()
  if (!user) {
    throw new Response('Unauthorized', { status: 401 })
  }
  return user.id
}
```

这个 `user.id` 是从**加密签名的会话 Cookie**里读出来的（由 better-auth 管理），用户**改不了、伪造不了**。然后每个查询都强制带上它：

```ts
// src/server/projects.ts —— 查询：只查自己的
return db
  .select()
  .from(projects)
  .where(eq(projects.userId, userId))
  .orderBy(asc(projects.sortOrder), asc(projects.createdAt))
```

注意连「更新」「删除」也都带了**双重条件** `id AND userId`：

```ts
// src/server/projects.ts —— 删除：id 对上但 userId 对不上，照样删不掉
await db
  .delete(projects)
  .where(and(eq(projects.id, data.id), eq(projects.userId, userId)))
```

这样即使张三猜到了李四某条数据的 `id`，因为 `userId` 对不上，也改不动、删不掉。

---

## 4. 为什么把逻辑「收口」到一个函数

如果每个查询都各自手写「从会话取 userId」的逻辑，那么多个文件、多个函数里，**只要有一处忘了过滤，就漏一个洞**。

把它统一成一个 `requireUserId()`，好处是：

- 所有业务函数都调同一个入口，要么登录了拿到 id，要么直接 401，没有第三种可能；
- 想改鉴权逻辑（比如加个封禁检查），只改一处；
- Code Review 时只要检查「每个查询是否带了 `eq(table.userId, userId)`」即可。

这也是为什么文档反复强调「UI 永远通过 `server/` 访问数据库，不直接 import `db`」——让隔离有**唯一入口**，不会被某个页面绕过。

---

## 5. 隔离 vs 索引：两个层面的事

- **隔离**是**安全**层面：保证读写只命中自己的数据。
- **索引**（`<table>_user_idx`，建在 `user_id` 上）是**性能**层面：因为全表混着所有用户的数据，每次查询都要 `where userId = ?`。没有索引时数据库要**扫全表**逐行比对；有索引就能直接定位到该用户的行，数据量大时差距巨大。

两者配合：隔离保证「对」，索引保证「快」。

> 设计取舍：业务表的 `user_id` **不建数据库外键**，仅在应用层强制过滤 + 加索引。这样避免 better-auth 表（CLI 托管）与业务表强耦合，迁移更自由。详见 `技术栈与架构.md` 第 4.1 / 4.4 节。

---

## 6. 业内都这么设计吗？（多租户对比）

「共享表 + `user_id` 隔离」是业内**最主流**的做法之一，但不是唯一。这背后是一个专门话题——**多租户架构（Multi-tenancy）**。通常分三种隔离模型，从「共享」到「独立」：

| 模型                                   | 做法                                            | 代表场景                                          | 优点                                   | 缺点                                       |
| -------------------------------------- | ----------------------------------------------- | ------------------------------------------------- | -------------------------------------- | ------------------------------------------ |
| **① 共享表 + 区分列**（本项目）        | 所有用户数据同表，靠 `user_id` / `tenant_id` 列区分 | 多数 SaaS 标配（GitHub、Notion、Slack、Linear 等） | 成本最低、运维最简单、加用户零成本、跨用户统计方便 | 隔离全靠应用层，**漏一行 `where` 就泄露**；单表数据量大 |
| **② 共享库 + 独立 Schema**             | 同一数据库实例，每租户一套独立 schema            | 中型 B2B SaaS（常见于 PostgreSQL）                | 隔离更强、单租户数据量小、可单独备份迁移 | 租户成千上万时 schema 管理成本爆炸         |
| **③ 独立数据库**                       | 每租户一个独立库甚至独立实例                     | 金融 / 医疗 / 政府等**强合规**，或高单价大客户     | 物理隔离最彻底，安全/合规/性能互不影响   | 成本最高、运维最复杂                       |

### 业内怎么选

| 业务类型           | 典型选择                       |
| ------------------ | ------------------------------ |
| 个人工具 / 小型 SaaS | **① 共享表 + user_id**（本项目） |
| 中大型 B2B SaaS    | 共享表为主，大客户单独拆库     |
| 强合规（金融/医疗）| 独立库 / 独立实例              |
| 超大规模           | 共享表 + **分库分表（sharding）** |

> 关键认知：很多大公司是**混合**的——绝大多数中小客户走「共享表」，少数付费高的大客户给「独立库」。这叫分层多租户。

### 本项目为什么选 ①

1. 个人/小团队工作台，用户量小，共享表的成本与运维优势最大化；
2. 数据库是 Cloudflare D1（一个 SQLite），本就不适合「每用户一个库」；
3. 已把隔离**收口到 `requireUserId()`** + `user_id` 索引，规避了共享表最大的软肋（漏写 `where`）。

### 共享表的「进阶版」（用户量暴涨后才需要，了解即可）

共享表模型不会被推翻，而是叠加这些技术：

- **行级安全（RLS，Row-Level Security）**：数据库层面强制按 `user_id` 过滤。即使应用层忘了写 `where`，数据库也会自动拦截——对共享表最有力的安全补强（PostgreSQL 原生支持）。
- **分库分表 / Sharding**：按 `user_id` 哈希把数据分散到多个库，解决单表过大。
- **表分区（Partition）**：按 `tenant_id` 做分区，提升大表查询性能。

---

## 一句话总结

> 多用户共用一张表 → 必须有个东西区分数据归属 → 这个东西（`user_id`）必须由**服务端从可信的登录会话**里取、并**强制**加进每一条增删改查的条件里。不这么做，用户就能看到甚至篡改别人的数据。
